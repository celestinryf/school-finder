## Step 3: Install mysql2

**Action:** `npm install mysql2`

**Decision:** `mysql2` (not `mysql`) because:
- `mysql2` has native Promise support (`mysql2/promise`) — no callback wrapping needed
- `mysql2` is faster (uses prepared statements by default)
- `mysql2` is actively maintained; the original `mysql` package is deprecated
- `mysql2` is the standard driver used by Prisma, Drizzle, and other ORMs under the hood

**Result:** Added 11 packages. Total: 371 packages, 0 vulnerabilities.

**What `mysql2` provides us:**
- `mysql2/promise` — Promise-based API for async/await usage
- `createPool()` — Connection pool management (reuses connections instead of creating new ones per query)
- Parameterized queries via `?` placeholders — SQL injection prevention
- Support for MySQL 8 authentication (`caching_sha2_password`)

| `DATABASE_PASSWORD` | MySQL password | Standard |

Individual vars are more explicit (aligns with "explicit over clever" preference) and easier to debug.

## Step 6: Database Connection Pool (`lib/db.ts`)

**Action:** Created `lib/db.ts` — 19 lines of code.

**Implementation details:**

```typescript
import mysql from "mysql2/promise";

let pool: mysql.Pool | null = null;

export function getPool(): mysql.Pool {
  if (!pool) {
    pool = mysql.createPool({
      host: process.env.DATABASE_HOST,
      port: Number(process.env.DATABASE_PORT) || 3306,
      user: process.env.DATABASE_USER,
      password: process.env.DATABASE_PASSWORD,
      database: process.env.DATABASE_NAME,
      waitForConnections: true,
      connectionLimit: 5,
      queueLimit: 0,
    });
  }
  return pool;
}
```

**Decisions and rationale:**

1. **Singleton pattern via module-level variable:** The `pool` variable is declared at module scope (`let pool: mysql.Pool | null = null`). The `getPool()` function checks if it exists; if not, creates it. In Node.js, modules are cached after first import, so this variable persists across all imports within the same process. In Vercel's serverless model, this means:
   - A "warm" function instance reuses the pool across invocations (good — no reconnection overhead)
   - A "cold start" creates a new pool (expected — unavoidable in serverless)

2. **`connectionLimit: 5`:** Railway's free tier likely has a low max connection limit (10-20). Setting our pool to 5 means even if multiple Vercel function instances are running, we're unlikely to exhaust Railway's limit. If we set this to 10+ and had 3 concurrent function instances, that's 30+ connections potentially.

3. **`waitForConnections: true`:** When all 5 connections are in use, new queries wait in a queue rather than immediately failing with an error. For a POC with low traffic, this is always the right choice.

4. **`queueLimit: 0`:** Unlimited queue length. A bounded queue would reject queries when full, which is useful for backpressure in production but unnecessary for a POC.

5. **`Number(process.env.DATABASE_PORT) || 3306`:** Environment variables are always strings. We convert to number with a fallback to MySQL's default port (3306). The `||` fallback handles both `undefined` (env var not set) and `NaN` (env var set to non-numeric value).

6. **Import style:** `import mysql from "mysql2/promise"` — We import the Promise-based API directly, not the callback-based `mysql2`. This allows `await pool.query(...)` syntax throughout the app.

7. **No connection test on startup:** The pool is lazy — it doesn't actually connect to MySQL until the first query is executed. This is intentional: if the database is down, we want the error to occur at query time (where we can return a 500 response) rather than at import time (which would crash the server).

## Step 7: Health Check API Route (`app/api/health/route.ts`)

**Action:** Created `app/api/health/route.ts` — 16 lines of code.

**Implementation:**

```typescript
import { NextResponse } from "next/server";
import { getPool } from "@/lib/db";

export async function GET() {
  try {
    const pool = getPool();
    const [rows] = await pool.query("SELECT 1 AS ok");
    return NextResponse.json({ status: "connected", result: rows });
  } catch (error: unknown) {
    const message = error instanceof Error ? error.message : "Unknown error";
    return NextResponse.json(
      { status: "error", message },
      { status: 500 }
    );
  }
}
```

**Purpose:** This route exists solely to verify the database connection works. Hit `GET /api/health` and you'll get either:
- `200 { "status": "connected", "result": [{ "ok": 1 }] }` — DB is reachable
- `500 { "status": "error", "message": "..." }` — DB connection failed with reason

**Decisions:**
1. **`SELECT 1 AS ok`** — The simplest possible query. It doesn't touch any tables, so it works even if the database is empty (schema not yet imported). It only tests that the connection pool can reach the MySQL server and authenticate.

2. **Error handling:** `error: unknown` with `instanceof Error` check. In TypeScript strict mode, caught errors are `unknown` type. We check if it's an `Error` to safely access `.message`. If not (shouldn't happen, but defensive), we return "Unknown error".

3. **No sensitive info in error response:** The error message from mysql2 may include the hostname or connection details. For a POC this is acceptable. In production, we'd sanitize this.

**Verification:** Build passes. The route appears as `ƒ /api/health` (dynamic, server-rendered on demand), confirming Next.js correctly identified it as a server-side API route, not a static page.

## Step 10: Greptile Code Review Fixes

**Context:** PR #1 was reviewed by Greptile (confidence 4/5). Four issues were flagged and fixed:

1. **`lib/db.ts` — HMR connection pool leak:** Module-level `let pool` was replaced with `globalThis._mysqlPool`. This survives Next.js hot module replacement in dev without creating new pools on each file change.

2. **`app/api/health/route.ts` — Error message leakage:** Raw `error.message` could expose DB host/port/credentials. Now only shows detailed errors in development; production returns "Internal server error".

3. **`sql/seed.mjs` — Hardcoded database name:** Verification query used hardcoded `ryf_celestin_and_sia_preston_db`. Now uses `process.env.DATABASE_NAME` with fallback.

4. **`app/layout.tsx` — Default metadata:** "Create Next App" title/description replaced with "School Finder" / "Search and explore colleges and universities".
